## F1: Dashboard 控制面板加载

**用户操作**: 打开浏览器访问 `http://localhost:8321/index.html`

**预期行为**:
1. 页面加载，显示完整的 Dashboard 布局
2. 左侧区域显示 noVNC Web 客户端（嵌入 iframe 或 canvas）
3. 右侧区域显示控制面板，包含所有操作按钮
4. 连接状态显示为 "未连接"（红色圆点）
5. UI 与原型图设计一致：
   - 标题栏: "BOSS直聘自动化管理系统"
   - 状态指示器: 连接状态（红/绿）+ 运行状态文字
   - 功能按钮区: 按顺序排列的功能按钮
   - 日志/输出区: 底部操作日志滚动显示

**视觉验收标准**:
- 页面加载无 4xx/5xx 错误
- 左侧存在 noVNC iframe 或 canvas 元素
- 右侧存在至少 5 个功能按钮
- 连接状态显示 "未连接" 文本
- 页面标题包含 "BOSS直聘"
- **视觉截图**: 与原型 layout 一致（左侧大面板 + 右侧窄面板）

## F2: 连接 VNC 远程桌面

**用户操作**: 点击右侧控制面板中的 **[连接VNC桌面]** 按钮

**预期行为**:
1. 前端发起 VNC 连接请求
2. noVNC 客户端连接 Docker 容器的 VNC Server（:1, port 5901）
3. 连接状态: "未连接" → "连接中..." → "已连接"（绿色圆点）
4. VNC 区域内显示 XFCE 桌面画面（壁纸、任务栏可见）

**视觉验收标准**:
1. 点击「连接桌面」按钮后，按钮显示 loading 状态
2. 左侧 VNC 占位符消失，noVNC iframe 全屏加载
3. 密码 `boss123` 通过 API 获取后自动拼入 noVNC URL 认证（无需手动输入）
4. XFCE 桌面完整渲染：可见任务栏、桌面图标、壁纸
5. 状态圆点从灰色变为绿色，文字显示 "已连接"
6. 浏览器 F12 控制台无 WebSocket 错误
7. 鼠标可在 VNC 桌面内移动和点击（交互可用）
- **视觉截图**: noVNC 区域可见完整 XFCE 桌面（任务栏 + 壁纸 + 图标）

## F3: 打开浏览器进入 Boss直聘网页

**用户操作**: 点击右侧控制面板中的 **[打开Boss直聘]** 按钮

**预期行为**:
1. 后端通过 Chromium 在 Docker 桌面内启动浏览器（`--user-data-dir=/app/data/chrome-profile`）
2. 浏览器自动导航到 `https://www.zhipin.com`
3. VNC 画面中出现 Chromium 浏览器窗口，显示 Boss直聘首页
4. Chrome profile 目录挂载到 Docker volume，容器重启后 cookie 保留

**视觉验收标准**:
1. 点击「打开BOSS直聘」按钮后，按钮显示 loading 状态
2. API 返回成功响应（200），含 `cdp_ready: true`
3. VNC 画面中出现 Chromium 浏览器窗口（窗口可见于 XFCE 桌面）
4. 浏览器地址栏或页面内容包含 "zhipin.com"
5. 页面显示 Boss直聘首页（不要求登录状态，首页加载即可）
6. 从点击按钮到 Chrome 窗口出现在 VNC 中 ≤30秒
7. 重复点击时复用已有 Chrome 窗口（检测已有进程 → 直接导航，不重复启动）
8. 无浏览器崩溃或错误页面
- **视觉截图**: VNC 中 Chromium 窗口可见 + zhipin.com 首页已加载

## F4: Boss直聘登录

**用户操作**: 点击右侧控制面板中的 **[检查登录状态]** 按钮（或人工在 VNC 中扫码）

**预期行为**:
1. 后端检测当前登录状态（通过 CDP 获取 cookie 或页面元素判断）
2. 如未登录：导航到登录页，VNC 中显示扫码二维码，用户用手机 BOSS直聘 APP 扫码
3. 如已登录：直接返回登录用户信息
4. 登录成功后持久化登录态：
   - **主方案**: Chrome `--user-data-dir=/app/data/chrome-profile`（挂载 Docker volume），cookie 自动保留
   - **备份方案**: 通过 CDP 导出 cookie 到 `data/cookies.json`，下次启动时注入回去
5. 容器重启后登录态仍有效

**视觉验收标准**:
1. API 返回成功响应，含 `logged_in: true/false`
2. 未登录时：VNC 中显示扫码登录页面（二维码可见）
3. 用户手机扫码成功后：页面跳转到已登录状态（显示用户头像/昵称）
4. 刷新页面仍为登录状态（cookie 有效）
5. API `/api/browser/check-login` 返回 `logged_in: true`
6. `data/cookies.json` 存在且非空（备份方案生效）
7. 无弹窗错误
- **视觉截图**: 已登录页面显示用户信息（头像/昵称/推荐页）

## F5: 批量主动打招呼

**用户操作**: 点击右侧控制面板中的 **[筛选+打招呼]** 按钮，设定上限人数

**预期行为**:
1. 自动化脚本导航到推荐牛人页面 `https://www.zhipin.com/web/chat/recommend`
2. 按筛选条件（学历/年限/学校白名单）全自动筛选候选人
3. 对符合条件者逐一点击 "打招呼"，发送预设招呼语
4. 全自动执行，无需人工介入
5. VNC 中可见浏览器持续操作：滚动列表 → 识别候选人 → 点击打招呼 → 发送消息
6. 底部日志区实时输出操作日志

**视觉验收标准**:
1. API 返回启动成功，后台任务开始运行
2. VNC 中可见推荐牛人页面（候选人卡片列表）
3. 连续截图（每 5 秒一张）可见页面状态变化（不同候选人被操作）
4. 日志区有实时打招呼输出（每条含候选人姓名 + 成功/跳过/失败）
5. 完成后日志区显示统计汇总（成功/跳过/失败数量）
6. 不出现浏览器崩溃或页面卡死
- **视觉截图**: 推荐牛人页面 + 候选人卡片 + 打招呼操作过程

## F6: 批量获取简历

**用户操作**: 点击 **[批量获取简历]** 按钮

**预期行为**:
1. 脚本识别聊天中对方已发简历的候选人
2. 过滤已获取过的简历（避免重复下载，查 `resume_operations` 表去重）
3. 对未获取者逐一点击查看/下载简历
4. 简历保存到本地 `data/resumes/` 目录
5. 后续需上传到飞书多维表格（Phase 2 扩展）

**视觉验收标准**:
1. API 返回启动成功
2. VNC 中可见候选人聊天详情页
3. `data/resumes/` 有新下载文件
4. 日志区有下载统计（含跳过重复记录）
5. `resume_operations` 表记录操作历史（去重依据）

---

## F7: 批量聊天回复

**用户操作**: 点击 **[批量回复未读]** 按钮

**预期行为**:
1. 脚本识别未读消息
2. 调用 DeepSeek API 结合以下上下文生成回复：
   - 对话历史（最近 10 轮）
   - 预设话术模板（用户在 Dashboard 中配置）
   - 知识提示词（岗位信息、公司介绍等）
3. AI 生成为主，话术模板作为兜底（AI 调用失败时使用模板）
4. 逐条发送回复

**视觉验收标准**:
1. API 返回启动成功
2. VNC 中可见聊天消息页面
3. 可见已发送的回复消息气泡（AI 生成内容）
4. 日志区有回复统计（含使用的生成方式：AI/模板兜底）
5. 话术内容合理、自然、无违规
6. `conversations` 表记录完整对话历史
